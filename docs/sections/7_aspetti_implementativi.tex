\section{Aspetti Implementativi}

\subsection{Struttura del Progetto}
Il progetto è organizzato secondo la struttura standard di un'applicazione Next.js con App Router. Di seguito i file e le cartelle principali:

\subsubsection{Lato Client (\texttt{app/})}
\begin{itemize}
    \item \texttt{app/layout.tsx} $\rightarrow$ Layout radice dell'applicazione, include i font e i metadata condivisi.
    \item \texttt{app/page.tsx} $\rightarrow$ Pagina di login comune per operatore e paziente.
    \item \texttt{app/(operatore)/layout.tsx} $\rightarrow$ Layout con sidebar per la vista operatore.
    \item \texttt{app/(operatore)/dashboard/page.tsx} $\rightarrow$ Dashboard con widget riepilogativi (pazienti attivi, alert, mood check).
    \item \texttt{app/(operatore)/pazienti/page.tsx} $\rightarrow$ Lista pazienti con ricerca e filtri.
    \item \texttt{app/(operatore)/pazienti/[id]/page.tsx} $\rightarrow$ Dettaglio paziente con anagrafica, stato clinico e contatti di emergenza.
    \item \texttt{app/(operatore)/attivita/page.tsx} $\rightarrow$ Interfaccia per assegnare esercizi fisici e cognitivi.
    \item \texttt{app/(operatore)/monitoraggio/page.tsx} $\rightarrow$ Grafici di aderenza terapeutica, risultati cognitivi e mood check.
    \item \texttt{app/(paziente)/layout.tsx} $\rightarrow$ Layout semplificato con icone grandi per la vista paziente.
    \item \texttt{app/(paziente)/home/page.tsx} $\rightarrow$ Piano giornaliero con card delle attività.
    \item \texttt{app/(paziente)/giochi/page.tsx} $\rightarrow$ Memory Card con feedback visivi e sonori.
    \item \texttt{app/(paziente)/mood/page.tsx} $\rightarrow$ Questionario umore con emoji.
    \item \texttt{app/(paziente)/esercizi/page.tsx} $\rightarrow$ Lista esercizi assegnati.
    \item \texttt{app/(paziente)/farmaci/page.tsx} $\rightarrow$ Conferma assunzione farmaci.
\end{itemize}

\subsubsection{Componenti Riutilizzabili (\texttt{components/})}
\begin{itemize}
    \item \texttt{components/Sidebar.tsx} $\rightarrow$ Barra laterale di navigazione per l'operatore.
    \item \texttt{components/PatientCard.tsx} $\rightarrow$ Card riepilogativa di un paziente.
    \item \texttt{components/MoodEmoji.tsx} $\rightarrow$ Componente interattivo con emoji per il mood check.
    \item \texttt{components/MemoryGame.tsx} $\rightarrow$ Logica e UI del gioco Memory Card.
    \item \texttt{components/ExerciseList.tsx} $\rightarrow$ Lista esercizi con stato di completamento.
    \item \texttt{components/DashboardChart.tsx} $\rightarrow$ Grafico riutilizzabile per la dashboard (Recharts).
\end{itemize}

\subsubsection{Lato Server (\texttt{lib/})}
\begin{itemize}
    \item \texttt{lib/supabase/client.ts} $\rightarrow$ Configurazione del client Supabase (browser).
    \item \texttt{lib/supabase/server.ts} $\rightarrow$ Configurazione del client Supabase (server-side, con cookie).
    \item \texttt{lib/actions/pazienti.ts} $\rightarrow$ Server Actions per operazioni CRUD sui pazienti.
    \item \texttt{lib/actions/attivita.ts} $\rightarrow$ Server Actions per assegnazione e gestione attività.
    \item \texttt{lib/actions/questionari.ts} $\rightarrow$ Server Actions per il mood check.
    \item \texttt{lib/actions/giochi.ts} $\rightarrow$ Server Actions per il salvataggio risultati giochi.
    \item \texttt{lib/schemas.ts} $\rightarrow$ Schemi di validazione Zod per i dati in ingresso.
    \item \texttt{middleware.ts} $\rightarrow$ Middleware per la verifica della sessione autenticata e redirect per ruolo.
\end{itemize}

\subsection{Sicurezza}

\subsubsection{Autenticazione}
Il sistema utilizza \textbf{Supabase Auth} per la gestione delle sessioni:
\begin{itemize}
    \item L'operatore accede tramite email e password.
    \item Il paziente accede tramite \textbf{magic link} o codice PIN numerico, pensato per la semplicità d'uso.
    \item Le sessioni sono gestite tramite JWT con refresh automatico. Il middleware Next.js verifica la validità ad ogni richiesta.
    \item Le password sono hashate con \textbf{bcrypt} dal servizio Supabase Auth.
\end{itemize}

\subsubsection{Row Level Security (RLS)}
La sicurezza a livello di dato è implementata con le policy RLS di PostgreSQL. Ogni tabella ha regole che garantiscono che un paziente possa vedere solo i propri dati e un operatore possa accedere solo ai dati dei pazienti a lui assegnati.

\subsubsection{Conformità GDPR}
\begin{itemize}
    \item \textbf{Minimizzazione dei dati:} si raccolgono solo i dati necessari.
    \item \textbf{Consenso informato:} al primo accesso il paziente conferma il consenso al trattamento.
    \item \textbf{Diritto alla cancellazione:} l'operatore può anonimizzare o eliminare i dati su richiesta.
    \item \textbf{Crittografia:} tutte le comunicazioni avvengono su HTTPS (TLS 1.2+). Supabase cifra i dati anche a riposo.
    \item \textbf{Istanza europea:} il progetto utilizza un server Supabase in regione EU.
\end{itemize}
