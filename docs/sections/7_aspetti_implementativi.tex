\subsection{Componenti React}

I componenti principali sono:

\begin{itemize}
    \item \textbf{\texttt{DashboardContent.tsx}:} gestisce ricerca, filtri e visualizzazione delle card paziente.
    \item \textbf{\texttt{PatientCard.tsx}:} mostra il riepilogo di un paziente nella dashboard.
    \item \textbf{\texttt{PazienteDettaglio.tsx}:} componente client piu' ricco del progetto. Gestisce tab, form, modali, farmaci, esercizi, grafici testuali e chiamata al suggerimento AI.
\end{itemize}

\subsection{Funzioni di supporto}

Durante il refactoring alcune parti ripetute sono state spostate in file piccoli dentro
\texttt{src/lib}. Questa scelta riduce duplicazione senza introdurre astrazioni difficili.

\begin{itemize}
    \item \textbf{\texttt{date.ts}:} contiene funzioni per ottenere la data corrente in formato ISO.
    \item \textbf{\texttt{form.ts}:} contiene letture sicure e semplici dei valori di \texttt{FormData}.
    \item \textbf{\texttt{metriche.ts}:} contiene il calcolo percentuale usato nelle metriche della dashboard.
    \item \textbf{\texttt{paziente-client.ts}:} recupera l'id del paziente collegato all'utente autenticato.
\end{itemize}

\subsection{Integrazione con Supabase}

Supabase viene usato per due responsabilita' principali:

\begin{enumerate}
    \item \textbf{Autenticazione:} login, registrazione, sessione e recupero dell'utente corrente.
    \item \textbf{Database:} lettura e modifica di \texttt{pazienti}, \texttt{farmaci}, \texttt{esercizi} e log giornalieri.
\end{enumerate}

Esempio semplificato di lettura dei pazienti dell'operatore:

\begin{verbatim}
const { data: pazienti } = await supabase
  .from('pazienti')
  .select('*')
  .eq('operatore_id', operatore.id);
\end{verbatim}

Esempio semplificato di registrazione dell'umore:

\begin{verbatim}
await supabase
  .from('umore_log')
  .upsert({
    paziente_id: pazienteId,
    data: dataOggi,
    valore: umore,
  });
\end{verbatim}

\subsection{Suggerimento AI}

Il suggerimento AI e' implementato in
\texttt{src/app/api/ai/suggerimento-operatore/route.ts}. La pagina di dettaglio invia solo dati
sintetici, come eta', stato del paziente, percentuale di aderenza ai farmaci, percentuale di
completamento esercizi e ultimo umore. La route chiama Groq tramite \texttt{fetch} e restituisce
una sola frase breve per l'operatore.

La chiave \texttt{GROQ\_API\_KEY} resta sul server e non viene inviata al browser. Per mantenere
il codice semplice, il controllo di accesso principale resta nella navigazione protetta e nelle
query Supabase usate dalle pagine.

\subsection{Sicurezza}

La sicurezza e' gestita principalmente su tre livelli:

\begin{itemize}
    \item \textbf{Supabase Auth:} controlla identita', login, logout e sessione.
    \item \textbf{Proxy Next.js:} protegge le pagine private e reindirizza gli utenti non autenticati.
    \item \textbf{Row Level Security:} limita l'accesso ai dati nel database in base al ruolo e all'associazione operatore-paziente.
\end{itemize}

Inoltre, le pagine server controllano i dati prima di mostrarli. Ad esempio, la pagina di dettaglio
paziente verifica che il paziente richiesto appartenga all'operatore autenticato.

\subsection{Scelte di semplicita'}

Alcune scelte sono state fatte per rendere il progetto piu' facile da spiegare e mantenere:

\begin{itemize}
    \item niente backend separato, perche' Next.js e Supabase coprono gia' le necessita' del progetto;
    \item niente librerie grafiche esterne, per mantenere lo stile controllato dal codice CSS;
    \item poche funzioni helper, solo dove riducono codice duplicato;
    \item route AI isolata, cosi' la logica di Groq non finisce dentro il componente React.
\end{itemize}
