\section{Modello Architetturale}

L'architettura di A.L.I.C.E. e' una web application full-stack basata su
\textbf{Next.js App Router} e \textbf{Supabase}. Il progetto non usa un backend
Express separato: la logica server viene gestita da Server Components, Server Actions,
Route Handlers e dal database PostgreSQL esposto da Supabase.

\subsection{Architettura hardware}

Il sistema e' pensato per un'infrastruttura managed. L'utente usa un browser, mentre
l'applicazione viene eseguita su piattaforma serverless e si appoggia a servizi esterni
per database, autenticazione e generazione del suggerimento AI.

\subsection{Modello architetturale}

Il modello architetturale adottato e' una combinazione di:

\begin{itemize}
    \item \textbf{architettura serverless}, per evitare la gestione manuale di un server applicativo;
    \item \textbf{Backend-as-a-Service}, per delegare a Supabase autenticazione, database, API REST e Row Level Security;
    \item \textbf{architettura component-based}, per costruire l'interfaccia con componenti React riutilizzabili;
    \item \textbf{pattern Server Action}, per gestire le modifiche ai dati direttamente lato server senza creare molti endpoint manuali.
\end{itemize}

Non si tratta quindi di un classico MVC con controller separati. Nel progetto:

\begin{itemize}
    \item le \textbf{viste} sono le pagine e i componenti React;
    \item la \textbf{logica applicativa} e' distribuita tra Server Components, Server Actions e helper in \texttt{src/lib};
    \item il \textbf{modello dei dati} e' lo schema PostgreSQL gestito da Supabase.
\end{itemize}

\subsection{DBMS}

Il DBMS e' PostgreSQL, fornito tramite Supabase. Il codice applicativo non apre connessioni SQL
dirette, ma usa il client Supabase per interrogare tabelle e funzioni. Le regole di accesso ai
dati vengono applicate anche a livello database tramite Row Level Security.

Le principali motivazioni della scelta sono:

\begin{itemize}
    \item schema relazionale adatto a pazienti, operatori, farmaci, esercizi e log;
    \item integrazione diretta con Supabase Auth;
    \item policy RLS per limitare i dati visibili a ogni ruolo;
    \item API REST generate automaticamente da PostgREST.
\end{itemize}
